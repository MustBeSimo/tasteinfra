07.5 — Implementation Blueprint

How to Build This: The Technical Stack

The Gatekeeper Model isn't theoretical. Here's how to implement each gate with specific tools and patterns.

GATE 1: SAFETY & COMPLIANCE
Tools: OpenAI Moderation API, Azure Content Safety, custom regex patterns
Pattern: Multi-layer filtering (API + custom rules + PII detection)
Golden Set: 500+ test cases across prohibited content categories
Performance: ~50ms latency, <0.1% false positive rate
Code Example: 
# Python implementation
from openai import OpenAI
import re

def gate_1_safety(text):
    # Layer 1: OpenAI Moderation
    response = client.moderations.create(input=text)
    if response.results[0].flagged:
        return {"pass": False, "reason": "content_policy"}

    # Layer 2: PII Detection
    if re.search(r'\b\d{3}-\d{2}-\d{4}\b', text):  # SSN pattern
        return {"pass": False, "reason": "pii_detected"}

    # Layer 3: Custom brand safety rules
    prohibited = ["guarantee", "promise", "refund"]
    if any(term in text.lower() for term in prohibited):
        return {"pass": False, "reason": "brand_safety"}

    return {"pass": True}
GATE 2: TECHNICAL CORRECTNESS
Tools: RAGAS, DeepEval, RAG verification against knowledge base
Pattern: Retrieval-augmented fact-checking + embedding similarity
Benchmark: Compare against ground truth database (product catalog, FAQ, docs)
Performance: ~200ms latency, 94% hallucination catch rate
Code Example: 
# Hallucination detection using RAG
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Pinecone

def gate_2_correctness(query, ai_response, knowledge_base):
    # Retrieve relevant facts from knowledge base
    embeddings = OpenAIEmbeddings()
    vectorstore = Pinecone.from_existing_index("kb", embeddings)
    docs = vectorstore.similarity_search(query, k=3)

    # Check if AI response is grounded in retrieved facts
    fact_checker = OpenAI()
    prompt = f"""
    Retrieved Facts: {docs}
    AI Response: {ai_response}

    Does the AI response contradict or hallucinate beyond the facts?
    Answer: YES (reject) or NO (pass)
    """
    result = fact_checker.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}]
    )

    return {"pass": "NO" in result.choices[0].message.content}

07b — Honest Limitations

The Cost of Control

Rigorous evaluation is not free. It is a strategic choice with real trade-offs.

Latency

Better judges "think" longer. Adding a robust LLM-as-a-Judge layer can add 500ms - 2s to your response time.

Verdict: Unfit for real-time voice; essential for async email/reports.

False Negatives

Strict gates kill serendipity. An overly conservative "Brand Gate" will reject creative outliers that don't fit the training distribution.

Verdict: You must tune "Temperature" for the Gate, not just the Generator.

Maintenance

Your "Golden Set" rots faster than your code. As models change (weekly), your benchmarks become obsolete.

Verdict: You need a dedicated "Eval Ops" owner. It's not a set-and-forget script.

The Commoditization Question

Won't evaluation get cheaper too? Yes, for basic checks. Syntax validation, factual retrieval, and policy compliance are commoditizing fast.

But: Consequence-weighted judgment (brand risk, legal exposure, domain expertise) remains expensive because it's context-specific and liability-bearing.

07c — Speed vs. Quality

Making It Fast: Hybrid Architectures

Rigorous LLM-as-Judge evaluation adds 500ms-2s latency. Unacceptable for voice assistants (<300ms), live chat (<1s), or code completion (<200ms).

The Constraint:
You can't run GPT-5.2 as a judge on every keystroke. You need a tiered architecture that balances speed, cost, and coverage.

Solution: Tiered Evaluation Architecture

TIER 1: Fast Lane (~50ms)
├─ Lightweight classifiers (distilled models: DistilBERT, TinyLlama)
├─ Regex/rule-based safety checks (keyword blocklists)
├─ Embedding-based brand similarity (cosine distance to brand centroid)
├─ Cached evaluations for common patterns (Redis lookup)
└─ Rejects: 40-50% of bad outputs

↓ Only if Tier 1 passes

TIER 2: Medium Lane (~300ms)
├─ Smaller LLM judges (Claude Haiku 4.5, GPT-4o-mini)
├─ Focused prompts for specific risks (factuality, tone)
├─ Parallel execution (safety + brand checks simultaneously)
└─ Rejects: additional 30-35%

↓ Only for high-stakes outputs

TIER 3: Thorough Lane (~1-2s)
├─ Full Claude Sonnet / GPT-5.2 evaluation
├─ Multi-step reasoning chains
├─ Only for customer-facing, high-risk, or legal content
├─ Async pattern: User sees "Verifying..." loading state
└─ Catches: remaining 10-15% of critical issues

Pattern: Risk-Weighted Routing

Query Risk Classification
  • Low-risk: Internal tools, drafts, brainstorming → Tier 1 only
  • Medium-risk: Customer emails, blog posts → Tier 1 + 2
  • High-risk: Legal docs, medical advice, financial → All 3 tiers
Real-World Performance
Baseline: Always-on full eval → 1,800ms P95 latency
Tiered: Risk-weighted routing → 280ms P95 latency
Coverage: 94% of critical issues caught
Cost: 60% reduction in eval compute

Advanced Optimization Techniques

1. Speculative Evaluation
Start eval before generation completes (streaming). Cancel if user stops typing.
2. Batch Processing
For async contexts (email, reports), batch 10-50 evals together for better throughput.
3. Progressive Disclosure
Show output immediately with confidence score. Upgrade to Tier 2/3 in background.
4. Semantic Caching
Cache eval results by embedding similarity. Similar inputs → reuse judgments (90%+ hit rate).
Emergency Fallback Pattern:
If all gates fail or timeout, show: "Let me check with a human colleague on this. I want to make sure I give you accurate information."

Graceful degradation > shipping unverified outputs.

08.5 — Operational Excellence

Measuring Success: The Eval Scorecard

You can't improve what you don't measure. Here are the KPIs that matter for evaluation infrastructure.

Core Metrics: The 6 Numbers You Must Track

1. Defect Escape Rate (DER)
Formula: Critical defects reaching production / Total outputs
Target: <0.5% (vs. 5-8% baseline without evals)
Why it matters: This is your north star. Every escaped defect is a potential incident.
2. Precision @ Gate
Formula: True positives / (True positives + False positives)
Target: >90% (avoid over-rejecting good content)
Why it matters: Low precision = wasting good outputs, frustrating users.
3. Recall @ Gate
Formula: True positives / (True positives + False negatives)
Target: >95% (catch the bad stuff)
Why it matters: Low recall = defects slip through, undermining trust.
4. Evaluation Cost per Output
Formula: (Infrastructure + Compute) / Outputs processed
Target: <20% of generation cost
Why it matters: Economics must work. Eval shouldn't 10x your AI bill.
5. P95 Latency
Formula: 95th percentile eval time
Target: <500ms for async, <200ms for sync
Why it matters: User experience. Slow evals kill product velocity.
6. Human Review Queue Size
Formula: Outputs flagged for manual review / Total outputs
Target: <5% (keep humans in loop, but efficient)
Why it matters: Scalability. 20% queue = you need a review army.

Technical Appendix

Deep Dive: Building Evaluation Systems

For technical teams ready to implement. This section covers reward model training, LLM-as-Judge patterns, and Golden Set construction.

Appendix A: Building Your First Reward Model

Architecture & Training Approach
Base Model: DeBERTa-v3-large or similar cross-encoder (350M params)
Training Method: Pairwise ranking using Bradley-Terry model
Output: Scalar quality score 0-1 (higher = better)
Typical Accuracy: 85-92% agreement with human experts after fine-tuning
Training Data Requirements
Minimum viable: 1,000 comparison pairs (A vs. B judgments)
Recommended: 5,000+ pairs for production quality
Collection method: Expert A/B judgments on real outputs from your domain
Cost estimate: ~$5-10K for quality labels (Scale AI, Surge AI, or internal experts)
Labeling time: ~30 seconds per pair with trained raters
Code Example: Training a Reward Model 
# Install: pip install sentence-transformers datasets
from sentence_transformers import CrossEncoder
from torch.utils.data import DataLoader

# Load your comparison data
# Format: [(prompt, output_A, output_B, label)]
# label = 1 if A > B, 0 if B > A
train_samples = [
    {'texts': [prompt, good_output], 'label': 1},
    {'texts': [prompt, bad_output], 'label': 0},
    # ... more pairs
]

# Initialize cross-encoder
model = CrossEncoder(
    'cross-encoder/ms-marco-MiniLM-L-6-v2',
    num_labels=1,  # Regression task
    max_length=512
)

# Fine-tune on your domain data
model.fit(
    train_dataloader=DataLoader(train_samples, batch_size=16),
    epochs=3,
    warmup_steps=100,
    evaluation_steps=500
)

# Save for production
model.save('models/quality-judge-v1')

# Inference: Score a new output
score = model.predict([[prompt, candidate_output]])[0]
if score < 0.65:  # Tune threshold based on precision/recall
    reject_output()
else:
    ship_output()
Pro Tip:
Start with 500 pairs to validate the approach. If accuracy >80%, scale to 5K pairs. If <70%, your labeling instructions are unclear or task is too subjective.

Appendix B: LLM-as-Judge Prompt Engineering

Pattern: Constitutional AI Principles
Define explicit evaluation criteria as "constitutional principles." The judge evaluates adherence to these principles. 
You are a brand quality judge for [COMPANY_NAME].

CONSTITUTIONAL PRINCIPLES:
1. Factual Accuracy: No claims without citation or verification
2. Tone: Professional, warm, never condescending or jargony
3. Structure: Clear headings, bullet points, <500 words
4. Legal Safety: No medical advice, no guarantees, no financial recommendations
5. Brand Voice: [YOUR_BRAND_VOICE_DESCRIPTION]

OUTPUT FORMAT (JSON):
{
  "score": 0-100,
  "passes": true/false,
  "reasoning": "Brief explanation of why it passes or fails",
  "failed_principles": ["principle_name"],
  "severity": "low/medium/high"
}

CANDIDATE OUTPUT TO EVALUATE:
{output}

YOUR EVALUATION:
Calibration Process
  • Test on 100 samples with known outcomes
  • Measure precision and recall at different score thresholds
  • Adjust threshold until Precision ≈ Recall ≈ 92%
  • Monitor for prompt drift (retest monthly)
  • Version your prompts (Git commit prompts like code)
Common Pitfalls
  • Vague criteria: "Good quality" → Define precisely
  • Too many principles: >7 principles = confusion
  • No examples: Add 2-3 pass/fail examples
  • Model bias: GPT-5.2 prefers longer outputs. Normalize for length.

Appendix C: Golden Set Construction

What is a Golden Set?
A carefully curated test suite of inputs with known expected outputs. Your "unit tests" for AI evaluation.

Purpose: Detect regression when you change prompts, models, or eval logic.
Construction Protocol
Step 1: Sample Real Traffic
  ├─ Collect 500-1000 representative inputs from production
  ├─ Include edge cases (ambiguous, adversarial, boundary conditions)
  └─ Cover full distribution (common queries + rare dangerous ones)

Step 2: Generate & Label
  ├─ Run through your AI system
  ├─ Expert humans label: PASS / FAIL / UNCERTAIN
  ├─ For each, record: Why pass? Why fail? Which gate should catch?
  └─ Discard "UNCERTAIN" cases (too ambiguous to test against)

Step 3: Version & Automate
  ├─ Store in Git (JSON/CSV with metadata)
  ├─ CI/CD runs Golden Set on every prompt change
  └─ Alert if pass rate drops >3%

Step 4: Refresh Quarterly
  ├─ Models change, attack patterns evolve
  ├─ Add 50-100 new cases per quarter
  └─ Archive obsolete cases (no longer representative)
Minimum Viable Golden Set
Total: 200 cases
• 100 PASS cases (should ship)
• 100 FAIL cases (should reject)
• Covers each gate (25 per gate)

Start here. Expand to 500+ for production.
Example Structure
{
  "id": "case_001",
  "input": "What's the refund policy?",
  "output": "We guarantee full refunds...",
  "expected": "FAIL",
  "gate": "safety",
  "reason": "Uses word 'guarantee'"
}
The Golden Rule:
Your Golden Set is only as good as your worst mislabeled case. If you're unsure, get a second opinion. Garbage in = false confidence out.
